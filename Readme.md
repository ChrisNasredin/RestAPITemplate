## Best Practices

### Конфиг

Конфиг сделан через flag --config.
Так же приложение поддерживает ENV, которые являются более приоритетными, чем параметры
в config.yml.

### Middleware:
#### Обработка ошибок в middleware
В моем самом первом проекте ошибки и их обработка/логирование - это была боль.
В данном темплейте ошибка просто из хэндлера пробрасывается наверх. Если ошибка
легитимная и не связана с тем, что что-то пошло не так - мы ставим соответствующий
статус, пишем пользователю, что случилось. Если ошибки нет в ожидаемых ошибках - 
Мы ставим статус 500, выдаем пользователю нейтральную "Internal Server Error" и 
в логи уже выплевываем все ошибку, причем с полным стеком вызовов. В константе _op_
в каждом методе описывается полный путь метода и ошибка оборачивается через _%w_
тем самым давая нам корректный стек-трейс. В логах это выглядит вот так:
```
transport.http-server.GetItem -> domain.Service.GetItem -> storage.postgres.GetItemByID -> Database Error
```

Потрясная штука, не нужно каждый раз писать простынь в каждом хэндлере.
Все ошибки определены в одном месте, лог тоже формируется в одном месте, если
он необходим. Делает хэндлеры чище.

Тут же обрабатываются ошибки из валидатора и отдаюся в нормальном виде, кроме того,
их можно кастомизировать при желании. 
Так же при желании можно отдать `ERR_CODE` для лучшей интеграции с клиентами.

Из минусов - нельзя просто так взять и встроить в middlewareChain из-за кастомного
типа хэндлера. Можно, конечно переписать все middleware под этот тип, возможно, 
позже я пойму, что именно так нужно было сделать и переделаю

#### Middleware RequestID

Очень прикольный паттерн - зашивается логгер в контекст запроса, причем в логгер
уже зашит request ID. В любой функции можно из контекста вытащить логгер и 
залогировать все, что нужно и это будет контекст того самого запроса.

#### Middleware logger

Ничего необычного, просто может логировать все запросы, что тоже достаточно
удобно. Из прикольного - переопределяется стандартный метод WriteHeader, 
чтобы вытащить статус. Логгер получаем из контекста запроса.

#### Panic Recovery

Очень простой, но довольно полезный middleware. Ставится обязательно самым первым,
чтобы отлавливать панику на любом уровне. Суть в том, что, в общем-то, ловит панику,
восстанавливает ее и отправляет лог. Лог пока сделан общий, не стал пока делать матрешку
для замыкания логгера - но глобальный логгер прописан как мой, так что не страшно

#### Middleware Chain

Это штука, чтобы у ревьюевера не вытекли глазки.
Чтобы не вкладывать матрешки и middleware были более читаемыми, написал вот такую штуку.
Это я написал сам, не знаю, насколько это бест-практис, но мне кажется, что вложенные
функции хоть и выглядят более наглядными в плане выполнения, но читаются сильно хуже,
чем цепочка.

Без chain:
```Go
mw := middleware1(middleware2(middleware3(log)(middleware4(http.Handler))))
```
С chain:
```Go
mw := middleware.Chain(
	middleware1,
	middleware2(log),
	middleware3,
)
```

### HTTP

Тут все довольно компактно упаковано в функции-хелперы `HandleBody` и `ResponseJSON`, что оставляет хэндлеры довольно чистыми.
Вся валидация, сериализация/десериализация происходит за кулисами
Все ошибки отдаются на откуп Middleware.
Как видно, вся работа строится вокруг доменной бизнес-сущности, то есть все строго - 
с базой работаем через StorageItem, c JSON - ItemResponse. Такой подход позволяет на засирать структуры
структурными тегами и в целом деражть слои разделенными - все зависит только от доменной сущности.

### Миграции

Миграции реализованы с помощью Goose. Миграции встроены в бинарник, чтобы не
тащить зависимость в контейнер. Запускаются флагом --migrate,
требуют пользователя с привилегиями уровня DDL.
Такую штуку будет удобно деплоить как в докер так и в k8s

### Observability

Минимальные метрики, отлавливаются через Middleware + пробы для k8s.
Ручки observability упрятаны в пакет, и инициализируются в main одной строчкой.
Почище, да и заодно показано, как можно спрятать добавление ручек, на случай, если в 
main.go будет мусорно. Кстати, middleware тоже можно добавлять в пакете, а не в main.
Метрики добавлять по необходимости.
Для примера добавлена бизнес-метрика `items_count` которая показывает количество активных
сущностей
